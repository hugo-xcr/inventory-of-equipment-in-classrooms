# InventoryOfEquipmentInClassrooms

Информационная система учёта и контроля перемещения оборудования в учебных аудиториях с функцией импорта данных.

Дипломный проект по специальности 09.02.07 «Информационные системы и программирование».  
Политехнический колледж НовГУ, 2026 г. Автор: Д.М. Савченко.

---

## Описание

Десктопное приложение для коменданта учебного корпуса. Заменяет бумажные журналы и таблицы Excel единой базой данных: ведёт реестр оборудования, фиксирует перемещения между аудиториями, автоматически формирует накладные и акты осмотра в формате .docx, поддерживает массовый импорт из Excel.

---

---



## Функциональность

- Ведение реестра оборудования с привязкой к аудитории и материально ответственному лицу
- Оформление постоянной и временной передачи оборудования между аудиториями
- Автоматическая генерация накладных на внутреннее перемещение (.docx)
- Оформление актов осмотра и списания с переводом объекта в архивный статус
- Массовый импорт оборудования из файлов Excel с предпросмотром
- Поиск и фильтрация по аудитории, ответственному лицу, инвентарному номеру, состоянию
- Управление справочниками аудиторий и преподавателей
- Просмотр истории передач по каждой единице оборудования
- Автоматическое определение единиц измерения и кодов ОКЕИ по наименованию

## Структура базы данных

| Таблица | Назначение |
|---|---|
| `job_title` | Должности сотрудников |
| `user` | Учётные записи: email, пароль, ФИО, должность |
| `room` | Аудитории и закреплённые преподаватели |
| `item_category` | Классификатор типов оборудования |
| `inventory_item` | Реестр оборудования: инв. номер, стоимость, статус, аудитория, МОЛ |
| `document` | Накладные и акты: тип, стороны, даты, статус |
| `document_item` | Позиции документа: связь с оборудованием, описание дефектов |

### Типы документов (`doc_type`)

| Значение | Описание |
|---|---|
| `waybill_permanent` | Накладная на постоянную передачу |
| `waybill_temporary` | Накладная на временную выдачу |
| `inspection_act` | Акт осмотра / списания |

### Статусы оборудования (`current_state`)

Управляются автоматически через триггер `trg_update_item_after_doc` при смене статуса документа на `Исполнен`:

- при `waybill_permanent` → `в наличии`, обновляются `room_id` и `custodian_id`
- при `waybill_temporary` → `во временном пользовании`, обновляются `room_id` и `custodian_id`
- при `inspection_act` → `списано`

---

## Триггеры базы данных

`trg_update_item_after_doc` — срабатывает перед обновлением поля `status` в таблице `document`. Если статус меняется на `Исполнен`, автоматически обновляет состояние всех позиций документа в таблице `inventory_item`. Это исключает ручное обновление данных на стороне приложения.

---

## Формат Excel-файла для импорта

Данные считываются начиная со строки 9. Используемые колонки: 3 — наименование, 10 — инвентарный номер, 13 — балансовая стоимость, 14 — количество. Строки с наименованием, начинающимся на `101.` или содержащим `Итого`, пропускаются.

## Контекст данных

Класс `DatabaseContent` (`Data/DatabaseContent.cs`) наследует `DbContext` и содержит все `DbSet<>` для работы с таблицами:

| DbSet | Таблица PostgreSQL |
|---|---|
| `JobTitles` | `job_title` |
| `Users` | `user` |
| `Rooms` | `room` |
| `ItemCategories` | `item_category` |
| `InventoryItems` | `inventory_item` |
| `Documents` | `document` |
| `DocumentItems` | `document_item` |

Доступ к контексту в приложении выполняется через статический метод `DatabaseContent.GetContext()` или через `using var dbContext = new DatabaseContent()` там, где требуется изолированная транзакция.

Маппинг настроен вручную в `OnModelCreating`: явно указаны имена таблиц, схема (`HasDefaultSchema`), внешние ключи и имя колонки `date_on_accounting` (во избежание обрезки имени драйвером).

## Архитектура

Приложение построено по трёхуровневой архитектуре: слой Windows Forms → слой бизнес-логики → Entity Framework Core → PostgreSQL. Модуль внешних интеграций (ClosedXML, Open XML SDK) вызывается из сервисного слоя после завершения транзакции.

---

## Тестирование

Реализованы три уровня тестирования:

- Unit-тесты бизнес-логики на InMemoryDatabase (добавление оборудования, передача)
- Функциональные тест-кейсы: авторизация (TU01), управление оборудованием (TU02), создание накладной (TU03)
- Интеграционные тесты связки UI — PostgreSQL — Open XML SDK

Все тест-кейсы имеют статус Pass.

---

## Безопасность

Пароли хранятся с хешированием BCrypt. Все запросы к БД выполняются через параметризованный EF Core (защита от SQL-инъекций). Внешние ключи с `ON DELETE SET NULL` и `ON DELETE CASCADE` обеспечивают целостность данных на уровне БД. Уникальность инвентарных номеров (`inventory_number UNIQUE`) и номеров аудиторий (`room_number UNIQUE`) контролируется ограничениями PostgreSQL.
